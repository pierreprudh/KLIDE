# v0.5 Agent Operations

## Status — 2026-07-15

The v0.5 operations foundation is shipped on `main`: Mission Control, shared
run lifecycle, review/evidence, worktree fleets, delegate live status, Git/PR
review, memory, and resumable transcripts are implemented. Post-v0.5 work now
includes two-agent races with side-by-side evidence comparison.

The release-candidate gap is operational hardening: a complete manual race and
restart/merge pass, provider-aware lifecycle markers for historical delegate
runs, automatic CI mileage, and a signed/notarized macOS bundle.

## Thesis

Klide should become the local-first operations surface for coding agents: start
runs, observe them, resume them, review their evidence, remember useful
outcomes, and safely let agents coordinate through Klide's trust boundary.

The Unpeel lesson is not "be a terminal manager." It is that users need one
calm place where many agent sessions keep living. Klide's answer is the
IDE-grade version: transcripts, diff review, permissioned commands, memory,
worktrees, and falsifiable evidence.

## Definition Of Done

v0.5 is done when a user can reopen Klide and confidently answer:

- What agent work is still live?
- What work needs my attention?
- What changed in the workspace?
- What can I safely resume, review, dismiss, or archive?

## Slice 1: Trust Fixes

These bugs come before new orchestration because they decide whether Klide feels
like durable infrastructure or a clever window.

- Persist API conversations reliably across app/window reopen, not only view
  switches. Fixed the fragile edge: panel sessions still win, but if
  `klide.panelSession.<panelId>` is missing or from an older build, `AiPanel`
  now restores the latest saved non-delegate conversation for the same
  workspace/provider instead of opening empty.
- Stop completed/dismissed in-conversation todos from repopulating after the
  user closes them. Fixed in `TodoStrip`: completed-plan dismissal is remembered
  per workspace/conversation/todo signature, while changed or resumed work still
  reappears.
- Clarify MLX tool support: either prove the current OpenAI-compatible tool
  path works, or label unsupported models as chat/plan only. Verified at the
  code seam: MLX advertises tool support, uses static presets, and the
  OpenAI-compatible parser accepts MLX's complete-message `tool_calls` shape.

## Slice 2: Run Lifecycle Contract

Normalize every work item, whether it comes from Klide's Harness or a Delegate,
into a shared lifecycle:

- `queued`
- `running`
- `waiting`
- `needs_review`
- `done`
- `failed`
- `cancelled`

The lifecycle must feed Mission Control, transcripts, memory drafts, delegate
adapters, and row-level actions. A run can be stale, archived, or superseded,
but it should not be forgotten accidentally.

Started: `RunLifecycleStatus` now projects raw adapter/transcript statuses into
the product contract (`error` becomes `failed`, completed delegated tasks become
`needs_review`). The run ledger carries both raw `status` and product
`lifecycle`, Mission Control's detail header shows the lifecycle label/color,
and stale running runs now produce an explicit idle attention reason.

Advanced: Mission Control now uses `lifecycle` for row decisions instead of raw
adapter status: attention dismissal keys, quick-send/retry actions, resume
affordances, failed-run danger states, evidence visibility, and the "memory not
saved" cue all read from the product lifecycle. Raw `status` remains available
for adapter-specific terminal/task controls.

Advanced: Klide Harness summaries now preserve live supervisor state in
`agent_list_runs`: `waiting_for_permission`, `waiting_for_diff`, and `paused`
are overlaid from the live run handle, mapped to Mission Control's `waiting`
lifecycle, and stale persisted active states are marked `cancelled` after a
restart. This prevents paused approval/diff runs from being buried as `done`.

Advanced: Klide-managed delegate PTYs now publish a backend live-session
contract (`status`, `updatedMs`, `startedMs`) instead of leaving Mission Control
to infer everything from a transcript timestamp. The live strip can distinguish
freshly active sessions from quiet-but-still-live sessions, sort by latest PTY
activity, and send exited delegated tasks through the `needs_review` lifecycle.

Still open: external on-disk delegate adapters still only have provider-specific
logs/databases, so Claude Code/Codex/OMP historical rows remain mostly
`running` vs `done` until we add provider-aware waiting/exit markers.

## Slice 3: Mission Control Dogfood Pass

Mission Control should answer the operator questions in under three seconds:

- group by state, not by implementation detail
- one obvious next action per row
- compact evidence: last event, worktree/branch, files touched, cost, memory,
  diff/review status
- stronger "needs me" detection for approvals, failed commands, idle delegates,
  and reviewable output
- less terminal noise in the primary row; terminal detail stays one click away

## Slice 4: Safe Delegate Session Control

Add Unpeel-style session control in Klide terms, not raw terminal puppeteering.

Start read-only:

- list project-scoped delegate sessions
- read latest output
- surface live/idle/needs-input signals

Then add explicit permissioned actions:

- send text
- answer a prompt/menu
- interrupt
- resume
- close/archive

Rules:

- every action emits a transcript event
- workspace/project scope is the default boundary
- cross-workspace control requires explicit user opt-in
- the Permission engine gates write-like or destructive session actions
- Mission Control remains the visible source of truth

## Slice 5: Goal Loop Supervision

Once delegate control is safe, Goal loops can supervise multiple runs:

> Fix failing checkout tests. Use Codex for patching, Claude for review, stop
> when tests pass.

The Harness still owns provider turns and tool execution. The Goal loop only
tracks the goal, gates, revision budget, active runs, evidence, and stop reason.
